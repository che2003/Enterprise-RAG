# `app.py` 完整页面需求设计（基于后端能力反推）

> 目标：将当前后端已实现的 **双路检索 + 可选文档隔离 + 按需模型冷启动 + 自动评测看板** 能力，完整映射为前端页面需求，便于后续 UI 重构或拆分为产品需求文档（PRD）。

---

## 1. 页面信息架构（IA）

### 1.1 全局区域
- 顶部品牌区
  - 标题：企业级 RAG 智能问答系统。
  - 副标题：强调 BBox 降噪切分、RRF 混合检索、LLM-as-a-Judge。
- 全局系统状态栏（始终可见）
  - 展示字段：
    - Embedding 引擎状态（就绪/失败）
    - LLM 生成引擎状态（就绪/失败）
    - 冷启动状态文本（进行中/完成/失败）
    - 检索索引状态（就绪/未就绪）
    - 索引摘要（chunk 数、文档数、耗时）
    - 大盘读取状态（CSV 读取结果）

### 1.2 主体分区（三标签页）
1. **智能交互中心（Chat）**
2. **私有知识库注入（Ingestion）**
3. **自动化评测监控大盘（Dashboard）**

---

## 2. 功能需求（按页面）

## 2.1 智能交互中心

### 2.1.1 控件与布局
- 左侧「检索控制台」
  - 检索范围下拉框：
    - 默认值：全局检索（混合所有文档）
    - 动态值：构建索引后追加每个上传 PDF 的文件名
  - 检索策略单选：
    - Method A（传统固定切分）
    - Method B（物理 BBox 降噪）
  - Top-K 滑块：范围 1~10，默认 5
  - 清空按钮：清空输入、会话、溯源内容
- 右侧主工作区
  - 对话窗口（Chatbot）
  - 问题输入框（回车发送）
  - 索引状态提示条（索引未就绪/构建中/就绪）
  - 折叠面板：底层召回证据（Chunks）

### 2.1.2 交互逻辑
- 提交问题时：
  1) 输入为空：前端提示“请输入有效问题”。
  2) 首次问答触发检索引擎按需初始化（DataPipeline + 双 Retriever）。
  3) 按策略调用 A/B 检索。
  4) 若用户选择了某文档，检索需带 `target_doc_name`，实现“防串库硬隔离”。
  5) 若命中 chunk：
     - 若生成模型可用：返回 LLM 生成答案
     - 若生成模型不可用：返回“检索摘要模式”降级文案
  6) 若未命中：返回“未检索到相关内容”。
  7) 溯源面板显示 fused context 文本。

### 2.1.3 异常与错误码展示
- 检索引擎初始化失败：`ENGINE_INIT_FAILED`
- 未建索引：`INDEX_NOT_READY`
- 检索运行时异常：`RAG_RUNTIME_ERROR`
- 前端必须支持：
  - 人类可读文案（短句）
  - 可展开 JSON 详情（错误码、message、action）

### 2.1.4 可用性要求
- 至少支持连续 20 轮问答不丢上下文。
- 输入提交到首 token 返回（或错误返回）< 5s（不含首次模型冷启动）。
- 清空按钮为幂等操作（重复点击无副作用）。

## 2.2 私有知识库注入

### 2.2.1 控件与布局
- 多文件上传控件（仅 `.pdf`）
- `chunk_size` 数值输入（默认 400）
- “解析并构建索引”按钮
- 实时日志面板（可滚动）

### 2.2.2 构建流程（串行可观测）
1. 参数校验
   - 无文件：提示并终止
   - chunk_size 非法：提示并终止
2. 引擎初始化
3. Method A 建库
   - 固定长度切分 + overlap
   - 构建 FAISS + BM25
4. Method B 建库
   - 版面 block 清洗与拼接
   - 构建 FAISS + BM25
5. 更新系统状态
   - `index_ready = True`
   - `index_summary = chunk数A/B + 文档数 + 耗时`
6. 将上传文档名同步到 Chat 页「检索范围」下拉

### 2.2.3 日志规范
- 日志需按阶段输出（收到文件、参数、A 构建、B 构建、完成/失败）。
- 每阶段有统一图标语义：
  - 进行中：⏳
  - 成功：✅
  - 失败：❌
  - 警告：⚠️

### 2.2.4 失败回滚策略
- 任一阶段失败：
  - `index_ready = False`
  - `index_summary` 写入失败原因
  - 引导用户先单文件重试

## 2.3 自动化评测监控大盘

### 2.3.1 控件与布局
- 刷新按钮
- 刷新状态框
- 指标 DataFrame 看板

### 2.3.2 数据加载需求
- 自动读取 `record/*.csv` 最新文件（按创建时间）
- 若文件不存在：展示“暂无评测数据”占位
- 若字段缺失：状态栏中报告缺失列但不阻断展示

### 2.3.3 指标定义与口径
必须展示以下六项（A/B 并列）：
1. Hit@5
2. MRR@5
3. Faithfulness（剔除“诚实拒答白卷”）
4. 智能拒答率
5. Relevance
6. ROUGE-L

“诚实拒答白卷”判定口径：`Faith=10 & Rel=0` 的样本视为拒答样本，
- 从 Faithfulness 平均值中排除
- 并用于计算拒答率

### 2.3.4 页面行为
- 页面加载时自动触发一次数据刷新。
- 用户手动刷新不应阻塞 UI（queue=False 行为）。

## 3. 全局状态机需求

## 3.1 状态对象（建议）
统一维护：
- embedding_ready
- model_ready
- engine_status
- index_ready
- index_summary
- dashboard_summary

## 3.2 状态迁移
- 初始态：未建索引、模型未必就绪
- 问答触发：检索引擎冷启动
- 首次需要生成时：LLM 冷启动
- 建库成功：索引就绪
- 刷新大盘：dashboard_summary 更新

## 3.3 一致性要求
- 任一功能更新状态后，系统状态栏必须同步刷新。
- 禁止同一字段出现“双定义覆盖”导致状态错乱（当前代码存在重复定义风险，重构时需消除）。

---

## 4. 非功能需求

## 4.1 性能
- 构建索引时提供进度日志，避免“假死感”。
- 对大 PDF 集，前端日志刷新频率建议 300~800ms。

## 4.2 可靠性
- 所有后端异常必须落到可展示文案。
- 错误信息支持“用户提示 + 工程调试详情”两层结构。

## 4.3 可维护性
- 页面事件绑定与业务函数分层：
  - UI 组件声明
  - 事件绑定
  - 业务 service（chat/build/load）
- 推荐将状态管理抽离成独立模块。

## 4.4 可扩展性
- 检索策略支持插件化（A/B/C）扩展。
- Dashboard 指标支持配置化映射（列名 -> 展示名 -> 计算口径）。

---

## 5. 验收清单（UAT）

1. 上传 2 份 PDF，点击建库，可看到 A/B 两阶段日志并最终索引就绪。
2. Chat 下拉可选择单篇文档；切换后不会召回其他文档 chunk。
3. 未建库时提问，出现 `INDEX_NOT_READY` 引导。
4. 模型不可用时，问答仍可返回降级文案，不崩溃。
5. 大盘无 CSV 时显示占位提示；有 CSV 时展示 6 个核心指标。
6. 点击刷新大盘后，系统状态栏的 dashboard_summary 同步变化。
7. 清空按钮可重入且不影响索引状态。

---

## 6. 建议的后续实现拆分（供 `app.py` 重构）

- `ui/layout.py`：页面结构与组件定义
- `ui/events.py`：事件绑定
- `services/chat_service.py`：问答流程与错误码
- `services/index_service.py`：建库流程
- `services/dashboard_service.py`：CSV 读取与指标计算
- `state/system_state.py`：统一状态对象与更新器

该拆分可显著降低 `app.py` 体积、减少重复状态定义风险，并提高测试可测性。
